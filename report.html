JC2503 Project Report: Real-time Multiplayer Puzzle Website

1. Website Design and Structure

This coursework implements a three-page website according to the JC2503 assignment brief: Introduction page (root), About page, and Game page. The Introduction and About pages present personal and academic information, while the Game page contains the interactive multi-user puzzle application. All pages use semantic HTML5 structure (header, nav, main, section/article, footer) and share the same navigation model so users can move between pages without losing context. The layout and style are implemented using one consistent CSS file under public/css/style.css, which supports desktop and mobile screen sizes.

The web server is built with Express and EJS templates. Static assets are served from the public directory, and routes are provided for /, /about, /game, and /report.html. The report is intentionally implemented as a standalone page, as required by the brief. The design prioritises usability over visual complexity: readable contrast, clear spacing, and predictable information grouping. On the Game page, the interface is organised into board, shared pool, score, and message areas, so turn-based status remains understandable during multiplayer play.

2. Game Application Logic and Socket.IO Communication

The game logic is server-authoritative and persistent in memory. The server stores the shared 4x4 board (16 cells), the pool of available blocks, active players, turn index, current turn assignment, and scoreboard. The block set contains exactly 16 unique combinations generated from 4 shapes and 4 colours. At game start the board is empty and all 16 blocks are in the pool, matching the assignment specification.

Turn handling follows strict cyclic order. When a turn starts, the server selects one block uniformly at random from the current pool and assigns it only to the active player. The active player can place that block in one empty cell; after placement, turn processing happens immediately. The server then checks contiguous lines of length 3 or more (horizontal, vertical, and both diagonals) for either equal shape or equal colour. All matched cells are removed, corresponding blocks are returned to the pool, and the acting player receives one point per removed block. If the board becomes full right after placement, the server clears the whole board, returns all blocks to the pool, and awards a 16-point jackpot to that player. The game never terminates by itself.

The multiplayer flow supports dynamic join and leave. New players are appended to the end of turn order. Disconnect or 60-second turn timeout is treated as departure, and turn order is recalculated so the game continues for remaining players.

Event Direction Purpose in Implementation
player-join Client to Server, then broadcast Register display name and append player to active queue.
player-leave Server to Clients Notify manual leave, disconnect, or timeout removal.
turn-start Server to Clients Announce current player and deadline for this turn.
place-block Client to Server, then broadcast Submit placement and sync move information to all clients.
clear-score Server to Clients Report cleared cells and awarded points.
board-full Server to Clients Report full-board clear and 16-point jackpot.
game-state Server to Clients Send synchronized full state (board, pool, players, scores, turn).

Validation is performed on every action request: only the active turn owner can place, only empty cells are legal, and invalid payloads are rejected with an error event.

3. Reflection: Challenges and Lessons Learned

The most challenging part was maintaining correct turn order under real-world multiplayer changes. A naive queue works in simple cases, but can break when a current player disconnects or when players are removed at different positions in the list. I solved this by centralising turn transition logic and recalculating the next valid index after each join/leave/timeout action.

Another challenge was implementing line clearing correctly. One placement can trigger more than one line at the same time (for example, horizontal and diagonal). The solution was directional scanning around the placed cell and merging results into a set so each block is counted at most once. This ensures scoring accuracy and avoids duplicate removals.

Overall, this project strengthened my understanding of full-stack web application design and the separation of responsibilities: server decides authoritative state and rules, while clients focus on rendering and user interaction. It also improved my debugging discipline in event-driven systems.
