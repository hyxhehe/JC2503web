Project Report: Real-Time Multiplayer Puzzle Game

Module: JC2503 Web Application Development | Format: Standalone report page

1. Overall Website Design

This project is a small website with three main navigation pages (Introduction, About, and Game) and one standalone report page. The public-facing pages are rendered through EJS templates and served by an Express server. Static assets, including styles and client-side game logic, are served from the public directory.

The report page is intentionally standalone and accessible directly through /report.html, without depending on the main site navigation flow. This keeps assessment content easy to reach in one URL while preserving the same visual language and readability approach used across the rest of the website.

2. Game Design

The game itself is a real-time, turn-based multiplayer puzzle on a 4 by 4 board. The core design principle is a server-authoritative state model. The server owns all game data (players, turn order, board cells, block pool, score, and timeout status), while the client is responsible for presentation and input. This separation reduces cheating risk, avoids state divergence between browsers, and keeps game rules consistent for all participants.

Gameplay uses a shared pool of 16 unique blocks built from four shapes and four colours. On each turn, one active player receives one random block from the pool and must place it on an empty cell before timeout. The score system rewards clearing lines of at least three matching blocks by colour or shape across horizontal, vertical, and diagonal directions. If a move fills the board, the board is cleared and the player receives a fixed jackpot.

3. Client-Server Communication and Socket.IO Event Handling

Socket.IO is used to support low-latency bidirectional communication. Event names are explicit, and each event has a clear responsibility. The communication pattern combines command events (client to server) and state/broadcast events (server to clients).

Event Direction Purpose
player-join Client -> Server Request to join with a validated display name.
leave-game Client -> Server Leave current controlled player when requested or page exits.
place-block Client -> Server Attempt to place the assigned block on one board cell.
game-state Server -> Client Authoritative synchronized snapshot including board, scores, pool, and per-client turn data.
turn-start Server -> All Clients Announce next active player and deadline timestamp.
player-join, player-leave Server -> All Clients Notify lobby/order changes and reasons (left, timeout, disconnect).
place-block, clear-score, board-full Server -> All Clients Broadcast move outcome, score updates, and full-board jackpot resolution.
error-message Server -> Client Return validation or rule errors such as duplicate name or invalid move.

The state synchronization strategy is snapshot-based. After each meaningful transition, the server emits updated state, and the client re-renders board, score list, pool visibility, and turn information. In addition, ephemeral events drive short UI feedback (for example message feed updates and placement highlights). This mixed strategy keeps logic simple: the snapshot guarantees correctness, while event-specific cues improve usability.

4. Reflection: Challenges, Trade-offs, and Lessons Learned

The main technical challenge was keeping multiplayer state coherent when users join, leave, timeout, or disconnect at arbitrary moments. In turn-based games, edge cases often appear in transition boundaries rather than in normal play. For example, removing a player who is currently active must update turn indices, clear timers safely, and immediately start the next valid turn without skipping users.

Another challenge was balancing user experience and implementation simplicity. Frequent full-state updates are straightforward and robust, but they can trigger repeated rendering logic on the client. During refinement, I improved maintainability by centralizing state emission and tightening timer lifecycle handling on the front end. These changes preserve behavior while reducing hidden complexity.

The most important lesson is that real-time applications benefit from explicit server authority and carefully named events. Separating permanent game state from temporary visual effects also makes debugging easier. Finally, small code quality improvements (deduplicating state send paths, preventing stale timers, and simplifying availability checks) can materially improve reliability without changing game rules.
