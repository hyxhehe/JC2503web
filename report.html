Project Report: Real-Time Multiplayer Puzzle Game

Module: JC2503 Web Application Development | Format: Standalone report page

1. Overall Website Design

The website contains three connected pages: Introduction, About, and Game. Introduction and About present personal content, while Game hosts the multiplayer application. The server uses Express with EJS templates and shared styling from public/css/style.css.

A standalone report page is provided at /report.html. It is intentionally outside the main navigation flow, matching the assignment requirement that the report should be directly accessible by URL when the server runs at localhost:8080.

2. Game Design and Logic

The game is a server-authoritative, turn-based multiplayer puzzle played on a shared 4×4 grid. The server stores all critical state: board cells, active player list, turn index, current turn assignment, scores, and available block pool. Clients only render state and submit user actions.

The block set contains 16 unique combinations from four shapes and four colours. At each turn, the active player receives one randomly selected block from the current pool and places it on an empty cell. After placement, the server checks contiguous lines of three or more blocks with matching shape or colour in horizontal, vertical, or diagonal directions. Matched blocks are removed, returned to the pool, and points are awarded per removed block.

If placement makes the board full, the board is cleared, all blocks return to the pool, and the active player receives a 16-point jackpot.

3. Client-Server Communication and Socket.IO Events

Socket.IO provides bidirectional, low-latency messaging for commands and broadcasts. Core command events from client to server are player-join, leave-game, and place-block. Core server events are game-state, turn-start, player-join, player-leave, place-block, clear-score, board-full, and error-message.

Event Direction Purpose
player-join Client -> Server Join the active list with validated, unique display name.
leave-game Client -> Server Leave one controlled player explicitly.
place-block Client -> Server Submit selected empty board cell during own turn.
turn-start Server -> All Clients Announce next player and turn deadline.
game-state Server -> Client Send authoritative snapshot (board, players, pool, current turn context).
clear-score, board-full Server -> All Clients Broadcast scoring outcomes and board/pool updates.
error-message Server -> Client Return invalid action feedback.

The synchronization approach is snapshot-first. After meaningful transitions, the server emits updated state for deterministic re-rendering. Event-specific broadcasts provide short-lived feedback (turn notices and score messages), while game-state remains the source of truth.

4. Reflection: Challenges and Lessons Learned

The most difficult part was reliable turn progression under frequent state changes: users can join, disconnect, timeout, or leave manually at any time. Correct handling requires synchronized updates of timers, turn index, and active-player arrays. If these updates are not consistent, the game can skip players or keep stale turns.

A second challenge was balancing clarity and responsiveness. Continuous full snapshots improve correctness and simplify debugging, but they increase render frequency on the client. To keep the user experience understandable, I separated persistent state rendering from temporary visual feedback such as placed-cell highlights and turn banners.

The main lesson is that real-time web applications are easier to maintain when authority is centralized on the server and event responsibilities are explicit.
