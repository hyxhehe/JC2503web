Real-time Multiplayer Puzzle Game

1. Overall System Design

This project implements a real-time multiplayer puzzle game using Node.js, Express, and Socket.IO. The system consists of three main pages: Introduction, About, and Game, with an additional standalone report page accessible at localhost:8080/report.html.

Technical Stack:

  • Backend: Node.js with Express framework
  • Frontend: EJS template engine with vanilla JavaScript
  • Real-time Communication: Socket.IO for WebSocket-based bidirectional communication
  • Routing: Express routes for serving static pages and game interface

2. Game Application Logic and Socket.IO Communication

The game logic is server-authoritative and persistent in memory. The server stores the shared 4x4 board (16 cells), the pool of available blocks, active players, turn index, current turn assignment, and scoreboard. The block set contains exactly 16 unique combinations generated from 4 shapes and 4 colours.

Turn handling follows strict cyclic order. When a turn starts, the server selects one block uniformly at random from the current pool and assigns it only to the active player. The active player can place that block in one empty cell; after placement, turn processing happens immediately. The server then checks contiguous lines of length 3 or more (horizontal, vertical, and both diagonals) for either equal shape or equal colour. All matched cells are removed, corresponding blocks are returned to the pool, and the acting player receives one point per removed block. If the board becomes full right after placement, the server clears the whole board, returns all blocks to the pool, and awards a 16-point jackpot to that player.

The multiplayer flow supports dynamic join and leave. New players are appended to the end of turn order. Disconnect or 60-second turn timeout is treated as departure, and turn order is recalculated so the game continues for remaining players.

3. Reflection: Challenges and Lessons Learned

Technical Challenges Encountered:

1. Turn Order Management: The most challenging part was maintaining correct turn order under real-world multiplayer changes. A naive queue works in simple cases, but can break when a current player disconnects or when players are removed at different positions in the list. I solved this by centralising turn transition logic and recalculating the next valid index after each join/leave/timeout action.

2. Line Detection Algorithm: Another challenge was implementing line clearing correctly. One placement can trigger more than one line at the same time (for example, horizontal and diagonal). The solution was directional scanning around the placed cell and merging results into a set so each block is counted at most once. This ensures scoring accuracy and avoids duplicate removals.

3. Real-time State Synchronization: Ensuring all players see the same game state required careful implementation of the emitGameState() function that broadcasts to all connected sockets with player-specific data filtering.

4. Input Validation: Server-side validation became crucial to prevent malformed requests and ensure game integrity, especially for block placement coordinates and player actions.

Key Learnings:

  • Socket.IO's room and namespace features enable efficient multi-player game architectures
  • Server-authoritative game logic prevents cheating and ensures consistency
  • Proper error handling and user feedback improve the multiplayer experience
  • Real-time applications require careful consideration of network latency and disconnection handling
  • Overall, this project strengthened my understanding of full-stack web application design and the separation of responsibilities: server decides authoritative state and rules, while clients focus on rendering and user interaction.
